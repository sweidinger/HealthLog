## TL;DR

HealthLog is a self-hosted personal-health PWA with AI insights, Withings + Apple Health ingest, and a doctor-report PDF. The decisions below are not negotiable inside the codebase ‚Äî they are how this project survives a fourteenth release without a rewrite. iOS code that contradicts any of them will fail review even when the diff is technically correct.

## The ten load-bearing rules

| # | Rule | Why |
| --- | --- | --- |
| 1 | Conservative semver ‚Äî patch beats minor | Releases are bugfix-heavy with additive features; the v1.5.0 ‚Üí v1.4.14 reroute confirmed users prefer many small patches over occasional minor jumps |
| 2 | Marc-Voice everywhere user-facing | Every artifact (commits, CHANGELOG, in-app copy, release notes) reads as Marc's authorship; never expose "Claude/AI/agent/marathon/phase" |
| 3 | AI insights are the differentiator | Visual, evidence-grounded, dynamic, multi-provider; not generic chat |
| 4 | Six locales from day 1 | EN + DE are Marc-maintained; FR / ES / IT / PL ship native first-party translation with a maintainership banner ‚Äî no machine-translated fallbacks |
| 5 | `develop` ‚Üí `main` via PR; tags on `main` only | Long-lived develop branch; main is release-only; GHCR + Coolify build only from main tags |
| 6 | No AI tells in user-facing copy | The user never sees the model name, the chain-of-thought, the "as an AI" disclaimer, the marathon phrasing, or the wave number |
| 7 | Multi-provider AI routing with failover | Anthropic ‚Üí OpenAI ‚Üí Codex ‚Üí local; one failing provider doesn't kill the Coach |
| 8 | Recharts stays | No Apple Charts swap, no Lightweight-Charts, no D3 rewrite ‚Äî visual identity is locked, replacement requires explicit approval |
| 9 | Postgres + OAuth, not SQLite + biometric | The server is a multi-user database, not a phone-shaped key-value store; iOS is a client of it |
| 10 | MDR 2017/745 boundary on medical features | Coach refuses GLP-1 dose recommendations (GROUND RULE 9) and drug-level estimates (GROUND RULE 15) with MDR + MDCG 2021-24 cites ‚Äî Research Mode is opt-in observational only |

## Rule 1 ‚Äî Conservative semver

| Symptom | Action |
| --- | --- |
| A release adds a new endpoint | Patch bump (`1.4.24 ‚Üí 1.4.25`) ‚Äî endpoint is additive |
| A release breaks an existing endpoint | Minor bump (`1.4.x ‚Üí 1.5.0`) ‚Äî only with deliberate migration |
| A release drops support for a measurement type | Major bump (`1.x.y ‚Üí 2.0.0`) ‚Äî never executed |

The v1.5 milestone exists for the iOS app + Apple Health rebase, not as a number-bump. v1.4.25 fits 90 commits past v1.4.24 baseline (insights sub-pages, GLP-1 end-to-end, six locales, source-priority two-axis) into a patch because everything is additive ‚Äî old clients keep working unchanged.

## Rule 2 ‚Äî Marc-Voice

| Channel | Voice rule |
| --- | --- |
| Git commit messages | English, terse, professional, no agent-voice |
| CHANGELOG.md | Marc-as-author, "Adds X" not "Claude added X" |
| GitHub release notes | Identical to CHANGELOG entry, no agent mentions |
| In-app copy (UI strings) | English-default with the rest of `messages/en.json`; same voice |
| Doc-site (`docs/`) | Marc-as-author |
| Audit reports under `docs/audit/` | Marc-as-author |
| `.planning/` working notes | Internal ‚Äî can be candid; never published |

A code review that finds `// generated by agent` or `// marathon phase X added Y` will reject the diff. v1.4.20 retroactively scrubbed the prior CHANGELOG to remove inherited voice tells; the rule is permanent.

## Rule 3 ‚Äî AI insights are the differentiator

The Coach drawer + per-status insight cards + Daily Briefing + correlation cards + weekly report are the selling point. They are:

- **Visual** ‚Äî every claim renders with a chart context, not a paragraph
- **Evidence-grounded** ‚Äî the model receives a real SNAPSHOT with the user's data, never invents numbers (the `---KEYVALUES---` sentinel exists to surface the load-bearing numbers verbatim back to the user)
- **Dynamic** ‚Äî fresh insights evict stale ones; mutations invalidate every read query touching the resource
- **Multi-provider** ‚Äî fallback chain is structural (see Rule 7), not "if Claude is down we panic"

iOS code that bypasses the snapshot to do "smart" local computation defeats the entire pipeline.

Pre-v1.4.25 the Coach used an English body with a `REPLY LANGUAGE: <locale>` footer. v1.4.25 flipped this to **native per-locale system prompts** with a YAML-encoded safety-contract matrix per locale (single source of truth across drafting + tests). The refusal-probe matrix in CI catches cross-locale regressions.

iOS strings live in the same `messages/{locale}.json` bundles. The `i18n-locale-integrity.test.ts` blocks any PR that drops a key in one locale without dropping it everywhere.

Daily work goes on develop. `main` only receives release merges and tags. The GHCR docker workflow builds **from main only** so a dev push to develop never poisons the production image. Coolify auto-deploy listens on the GHCR tag.

## Rule 6 ‚Äî No AI tells

Forbidden in user-facing surfaces:

| Phrase | Replacement |
| --- | --- |
| "As an AI‚Ä¶" | (delete the entire disclaimer line) |
| "I'm Claude / I'm GPT-4 / I'm an LLM" | (delete) |
| "marathon / phase / wave / session / agent" | Rephrase or delete |
| "I was trained on‚Ä¶" | (delete) |
| "I cannot help with that, however‚Ä¶" | A real refusal: "This is medical advice that needs a physician ‚Äî open the doctor report and bring it to your appointment." |

The Coach refusal copy is hand-written per locale in `messages/{locale}.json` under the `coach.refusal.*` keys; the prompt-pattern guard re-emits the localised refusal without ever hitting a provider, so the LLM cannot leak its own identity even under prompt injection.

## Rule 7 ‚Äî Multi-provider AI routing

Order of fallback (see `src/lib/ai/provider-runner.ts` ‚Üí `runRawCompletionWithFallback()`):

1. **User-configured providers** in their saved priority order (Settings ‚Üí AI Provider)
2. **Admin-configured fallback** (typically the operator's OpenAI key)
3. **No-key fallback** ‚Äî rule-based copy, never claims to be the AI

Each attempt is logged as a Wide-Event with `provider`, `attempts`, `firstStatus`. v1.4.25 W5 split provider-down from rate-limited: every-attempt-429 surfaces the `coach.provider.rate_limited` copy with a warning toast; all-attempts-failed surfaces `coach.provider.unavailable`.

iOS calls `/api/insights/chat` and reads `error.code` from the SSE `error` frame to render the right copy ‚Äî the iOS app never picks the provider directly.

## Rule 8 ‚Äî Recharts stays

The visual identity of every chart in the app is locked. Pre-v1.4.20 there was a Lightweight-Charts evaluation that got reverted for accessibility (no aria-labels, no high-contrast tokens). The chart library swap is in the "explicit-approval-required" category ‚Äî design + product sign-off before any PR.

For iOS, this means **Swift Charts is the equivalent of Recharts** ‚Äî pick the OS-native one, never roll a custom canvas renderer. The visual aesthetic (axis tokens, color rules, legend placement) is what the user remembers ‚Äî the renderer is what makes it stable.

## Rule 9 ‚Äî Postgres + OAuth, not SQLite + biometric

Server-side decisions:

| Choice | Rejected | Why |
| --- | --- | --- |
| Postgres | SQLite | Multi-user from v1.0; concurrent writes from Withings webhook + iOS batch + manual web entry |
| OAuth + session cookies + Bearer tokens | Biometric-only | Cross-device, web-first, audit-friendly |
| Argon2 password hashing | bcrypt / PBKDF2 | OWASP 2024 recommendation; libargon2 is mature |
| Passkeys via SimpleWebAuthn | Apple-only WebAuthn | Cross-browser support |

For iOS:

- The phone keychain stores the **refresh token**, not the user's password
- Apple Health is a **client-side data source**, not a primary store ‚Äî every HealthKit sample flows through `POST /api/measurements/batch` into the canonical Postgres
- Biometric unlock is a **UX shortcut** for accessing the locally-cached refresh token, not a security primitive

## Rule 10 ‚Äî MDR boundary

Two GROUND RULES make this binding inside the Coach prompt:

| Rule | Cite | Behaviour |
| --- | --- | --- |
| GROUND RULE 9 | EU MDR 2017/745 + MDCG 2021-24 | Coach refuses GLP-1 dose recommendations; offers EMA EPAR titration ladder as observational reference only |
| GROUND RULE 15 | EU MDR + EMA EPAR + Schneck/Urva 2024 | Coach refuses drug-level estimates; Research Mode chart is opt-in, behind a versioned acknowledgment dialog (Migration 0058), observational use only |

Research Mode lifecycle: `GET / POST / DELETE /api/auth/me/research-mode`. The acknowledgment carries a version string (`RESEARCH_MODE_DISCLAIMER_VERSION`); a stale version forces re-prompt. The drug-level chart is a one-compartment PK approximation ‚Äî qualitative trace only, no numeric "your current level is X ng/mL" claim.

iOS replicates this verbatim:

- The GLP-1 detail screen surfaces the Research Mode toggle in Settings ‚Üí Advanced
- Tapping the chart on a non-acknowledged account presents the dialog with the **live** server-side disclaimer copy (fetch via `GET /api/auth/me/research-mode`, do not hardcode)
- Re-prompt is automatic when the server's `currentDisclaimerVersion` differs from the user's `acknowledgedVersion`

## Cross-cutting consequences

### Cache invalidation on new content

A successful mutation invalidates every read query touching the resource. TanStack Query keys are namespaced per resource (see `src/lib/query-keys.ts`). The convention: same `queryKey` MUST map to the same unwrap shape ‚Äî `(await res.json()).data`. Two endpoints accidentally sharing a key but returning different shapes is a cache-poisoning bug that hides until production.

For iOS, the equivalent rule: every mutation that writes a measurement / workout / medication / mood MUST trigger a refetch of the matching list query. If the iOS state layer uses a different cache model (e.g. SwiftData), preserve the rule: never serve stale data after a write.

### No personal data in user-facing artifacts

Marc's name, BD-Zielbereich values, measurement counts, lab values ‚Äî none of these appear in CHANGELOG / GH releases / `docs/audit/v*-summary.md` / docs site / landing. The retroactive scrub of v1.4.14‚Äìv1.4.19 (in v1.4.20) and the pre-release PII-grep gate enforce this. For iOS: never put user data into commit messages, release notes, or screenshots checked into the repo.

### Umlauts everywhere

UTF-8 end-to-end. `"Nrnberg"` instead of `"N√ºrnberg"` is a bug. iOS string-handling must preserve umlauts through every encoding step; this includes URL parameters (use `URLComponents`, not string concatenation) and POST body bytes (always send `application/json; charset=utf-8`).

### Research before complex features

AI / insights / chart work runs through a research agent first. The agent benchmarks Apple Health / Withings / Oura, proposes an ecosystem-fit plan, then the implementation phase starts. The research artifacts live under `.planning/research/`. For iOS, mirror this: before building the Coach drawer or the dashboard, read the equivalent research note under `.planning/research/`.
