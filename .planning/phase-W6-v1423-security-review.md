# Wave 6 — Security review (v1.4.23)

Branch: `develop` (32 commits ahead of `main`).
Scope: every commit between `main` and `develop` for v1.4.23, focusing on the eight surfaces called out in the audit prompt (`POST /api/measurements/batch`, `POST /api/devices` APNs acceptance, APNs sender, device-management endpoints, refresh-token per-device revocation, `/api/auth/me/coach-prefs`, Coach message feedback, OpenAPI spec, Coolify runbook).

## Summary

- New attack surface in v1.4.23: batched Apple Health ingest with composite dedup, APNs token registration with cross-user hijack guard, an APNs sender that loads a `.p8` signing key from env, three device-management routes (list + delete via `/api/auth/me/devices` plus the iOS-friendly mirror at `DELETE /api/devices/[id]`), per-device scoping of refresh-token reuse-detection (legacy `null-deviceId` falls back to revoke-all), per-user Coach prompt-tuning prefs, per-message Coach feedback writing into the polymorphic `RecommendationFeedback` table, the auto-generated OpenAPI 3.1 spec at `docs/api/openapi.yaml`, and a maintainer runbook for Coolify auto-deploy at `.planning/coolify-auto-deploy-howto.md`.
- Findings: **0 CRITICAL · 0 HIGH · 2 MED · 4 LOW**.
- Overall posture: solid. The batch route is well-defended (entry cap, composite unique index, owner-scoped pre-flight + recheck, idempotency wrap). The cross-user-hijack guard is duplicated at both `token` and `apnsToken` layers and audit-logged on denial. Both DELETE-device routes return 404 (not 403) on cross-user attempts so device-id enumeration is blocked. The Coach feedback route enforces ownership via the `conversation: { userId }` join, rate-limits at 60/h, never echoes the prose back to admin dashboards, and dedupes via the existing unique index. The refresh-token reuse-detection legacy fall-back to user-wide revoke is preserved + tested. The OpenAPI spec exposes only the 9 iOS-touched routes — no `/api/admin/*`, no `/api/internal/*`, no debug endpoints. The two MEDs are about a missing DB-level uniqueness constraint on `apns_token` and the documented Coolify webhook URL embedding the production application UUID; the four LOWs are a per-entry status mismatch on intra-batch duplicates, a cached 422 that may surprise a retry author, an APNs key-file path leak in a warning string, and an audit row missing the userId on refresh-token failure.

## Findings

### MED-1 — `apns_token` has no DB-level uniqueness constraint, only an app-layer guard

- File: `prisma/migrations/0037_apns_device_columns/migration.sql:30` (creates `devices_apns_token_idx` as a plain index, not UNIQUE) + `prisma/schema.prisma:836` + `src/app/api/devices/route.ts:90-108` (the cross-user-hijack guard).
- What: the guard is a `findFirst({ where: { apnsToken, NOT: { userId } } })` followed by an upsert. Two concurrent `POST /api/devices` calls from two different users — both supplying the same APNs token — could each pass the lookup (no row yet) and then each insert a row, leaving two `Device` rows holding the same `apnsToken` for two different users. The `sendViaApns(userId, payload)` lookup at `src/lib/notifications/senders/apns.ts:319-329` filters by `userId`, so each user would still see their own row, but **the dispatcher would push the same APNs payload to the device for both users** — a redirected push.
- Why: APNs tokens aren't a secret, but they ARE device-stable, so the race window is theoretical (a user would have to physically share the token with another user and they'd both have to register within the same tick). Calling MED only because the CLAUDE.md commitment ("A device `token` belongs to exactly one user") is not enforced at the storage layer where it would be airtight.
- Fix: change the migration to `CREATE UNIQUE INDEX "devices_apns_token_unique_idx" ON "devices" ("apns_token") WHERE "apns_token" IS NOT NULL;` (partial unique index — multiple `NULL` values are still allowed for non-iOS device rows) and add `@unique` to the schema.prisma column. The application guard then becomes defence-in-depth: the unique index is the source of truth, and a P2002 thrown on the upsert maps cleanly to the existing 409 response.

### MED-2 — Coolify auto-deploy runbook embeds the live production webhook URL

- File: `.planning/coolify-auto-deploy-howto.md:25-26`.
- What: the runbook documents `COOLIFY_WEBHOOK = https://apps-01.bombeck.io/api/v1/deploy?uuid=pg8wggwogo8c4gc4ks0kk4ss&force=false` as the value the maintainer should paste into GitHub Secrets. That URL is the **actual live production deploy webhook**, not a placeholder. The same UUID also lives in 9+ historical audit / planning files (`docs/audit/v1414-summary.md`, `docs/audit/v1415-auto-deploy.md`, `docs/audit/v1416-auto-deploy-fix.md`, `docs/audit/v1418-summary.md`, `docs/audit/v1420-summary.md`, `.planning/phase-A1-report.md`, etc.).
- Why: triggering the webhook requires the `COOLIFY_TOKEN` Bearer in the `Authorization` header (`.github/workflows/docker-publish.yml:153`), so the URL alone cannot deploy. But the URL leaks (a) the Coolify host (`apps-01.bombeck.io`), (b) the API path (`/api/v1/deploy`), and (c) the production application UUID. Together with any future Coolify CVE that bypasses bearer-auth on `/api/v1/deploy?uuid=`, this becomes a one-click drive-by deploy of an attacker-controlled image. The MED rating reflects the historical exposure in `docs/audit/*.md` (already published in past releases) more than the new W4 runbook itself — v1.4.23 isn't introducing a new disclosure, just re-stating one. Worth fixing once for the whole repo.
- Fix: replace the literal URL in the runbook with a placeholder format (`https://<coolify-host>/api/v1/deploy?uuid=<your-app-uuid>&force=false`) and update the audit-summary docs in a future hygiene PR. The runbook should state "find this URL in Coolify UI → Application → Webhooks tab" without baking in the real value. Treat the UUID as a low-grade secret (deploy-trigger identifier) for future docs.

### LOW-1 — Intra-batch duplicate entries report `inserted` for both rows but only insert one

- File: `src/app/api/measurements/batch/route.ts:163-261`.
- What: when a single batch contains two entries with the same `(type, externalId)` pair — both not yet in the DB — the pre-flight `existing` lookup misses both, both get marked `inserted` in the per-entry results, then `createMany({ skipDuplicates: true })` inserts only one row. The reconciliation pass at line 226-261 only downgrades entries whose key is **NOT** in the recheck `stored` set; for an intra-batch duplicate the key IS in `stored` (because the one surviving insert is there), so neither entry is downgraded. Net result: `inserted: 1, duplicates: 0` in the count totals but `entries[*].status: "inserted"` for **both** indexes. The iOS sync cursor would then not flag either as a duplicate.
- Why: a self-inflicted accuracy bug at worst — the user can only confuse their own sync cursor. No data leak, no cross-user impact. The composite unique index still guarantees no duplicate row lands. Calling LOW because the per-entry status is the contract the iOS client uses to advance its cursor accurately, and the route's docstring at line 12-19 explicitly promises it.
- Fix: deduplicate `prepared[]` by `${type}::${externalId}` before the `existing` lookup. For each duplicate-within-batch, keep the first occurrence as `inserted` and mark every subsequent occurrence as `duplicate` immediately. Add a regression test in `tests/integration/measurements-batch.test.ts` covering "same externalId twice in one batch returns 1 inserted + 1 duplicate".

### LOW-2 — `withIdempotency` caches the 422 "batch too large" response

- File: `src/lib/idempotency.ts:131-138` (`isCachableStatus` allows 422) + `src/app/api/measurements/batch/route.ts:80-92`.
- What: per the documented contract ("validation 4xx IS cached so a broken retry doesn't re-hit the DB"), a 422 with `errorCode: coach.batch.too_large` from the entry-cap check is cached for 24h against the `(userId, key, method, path)` tuple. A retry author who first sent 600 entries (got 422), then trimmed to 500 entries and retried with the **same** `Idempotency-Key`, would silently get the 422 back without the second batch ever executing.
- Why: this is exactly the documented behaviour — Idempotency-Key contracts treat the body as immaterial after the first response. The intent is to prevent a broken-retry storm. Calling LOW because the 422-cache for "too large" is more confusing than a 422-cache for "missing field" — the body size is something the caller might naturally adjust between retries, whereas a missing field is a deterministic schema mismatch.
- Fix: leave the cache contract alone; instead, add a one-line note to the `coach.batch.too_large` error message reading "Idempotency-Key replay returns the original 422; retry with a fresh key." Optionally, mention the same in the OpenAPI `description` for the 422 response on `/api/measurements/batch`.

### LOW-3 — APNs key-file read failure surfaces the absolute path in a warning event

- File: `src/lib/notifications/senders/apns.ts:155-162`.
- What: when `APNS_KEY_FILE` is set but unreadable, the sender annotates the wide event with `getEvent()?.addWarning(\`APNs key file read failed: ${message}\`)`. Node's `ENOENT`/`EACCES`error messages embed the full filesystem path (e.g.`ENOENT: no such file or directory, open '/var/secrets/AuthKey_ABC1234.p8'`), so the path lands in the structured log payload.
- Why: low impact — operator-only observability, the path is the operator's own host filesystem, and the same warning is exactly what an operator wants when troubleshooting. Calling LOW because the wide-event payload also flows to Loki Push if `LOKI_*` env is configured, and a Loki tenant credential leak would then expose the path. The signing key itself is **not** in the warning (the failure is "couldn't even open the file"), so this is path-disclosure, not key-disclosure.
- Fix: replace `${message}` with a redacted variant — e.g. annotate `apns.config.key_file_unreadable` with `meta: { errno: err.code }` instead of the raw message. The `errno` is enough for an operator to triage (ENOENT vs EACCES) without leaking the path.

### LOW-4 — `auth.token.refresh.failed` audit row omits `userId` even when it can be resolved

- File: `src/app/api/auth/refresh/route.ts:68-83` + `src/lib/auth/refresh-token.ts:107-148`.
- What: `rotateRefreshToken` looks the row up by `tokenHash`, so on a `not_found` / `expired` / `revoked` / `already_used` failure it has the `userId` available (except `not_found`, which truly doesn't). The route at line 69 writes `auditLog("auth.token.refresh.failed", { ipAddress: ip, details: { reason: result.reason } })` without a `userId`, even when `result.reason === "already_used"` and the row that triggered the reuse-detection cascade is known.
- Why: refresh-token reuse-detection is **the** signal that a refresh token was leaked / replayed. Without `userId`, the audit-log dashboard can't pivot from a reuse event to "which user got their token reused" without re-querying the refresh-token table by hash. Calling LOW because the wider revoke (per-device or per-user) was already done by `rotateRefreshToken`, and the affected user's other devices got revoked, so the operational impact is contained — but reuse-detection deserves a high-fidelity audit trail.
- Fix: extend `RotationResult.failure` shape to carry the resolved `userId` when known (i.e., for `expired` / `revoked` / `already_used` — `not_found` truly has no user). Pass it through to the route's `auditLog` call. Same fix for the `revoke` path at line 45-49 (`revokeRefreshToken` already looks up the row, can return the userId alongside the boolean).

## Strongest defensive postures (v1.4.23 added)

- **Cross-user hijack guard, fully defence-in-depth.** Both `POST /api/devices` (`token` upsert) and the new `apnsToken` acceptance enforce the same boundary: a 409 on re-registration under a different user, with `device.register.denied` audit rows carrying the reason (`token_owned_by_other_user` / `apns_token_owned_by_other_user`) and the offending device id for incident-response triage (`src/app/api/devices/route.ts:90-126`). The pattern survives across both DELETE routes too — `/api/auth/me/devices/[id]` and `/api/devices/[id]` both return 404 (not 403) on cross-user attempts so device-id enumeration is blocked, and both routes scope refresh-token + access-token revocation to `deviceId`-only (not user-wide) so revoking one device never collateral-damages another (`src/app/api/auth/me/devices/[id]/route.ts:41-67`).

- **Refresh-token reuse-detection scoped per-device, with a safe legacy fall-back.** Pre-v1.4.23 a single replayed refresh token revoked **every** active refresh+access pair the user owned across every device, signing them out of their iPad when their iPhone replayed a token. v1.4.23 narrows the blast radius to the originating `deviceId` (`src/lib/auth/refresh-token.ts:117-148`), but tokens issued before v1.4.23 carry `deviceId === null` and therefore still trigger the wider revoke — there's no way to safely scope them, so the safety hatch is correct. The legacy null-deviceId path is unit-tested at `src/lib/auth/__tests__/refresh-token.test.ts:284`. The OpenAPI description at `src/lib/openapi/routes.ts:374-376` documents the per-device behaviour for the iOS team. The replay-detection error message ("Refresh token reuse detected — please log in again.") is identical regardless of which device triggered it, so there's no leak about which device the replay came from.

## Things uncertain — flagged for maintainer review

- **`/api/auth/me/devices` GET returns the full `Device.token` shape via the `web_push` channel inference, not the token itself.** The DTO at `src/app/api/auth/me/devices/route.ts:42-62` exposes `id`, `label`, `platform`, `appVersion`, `locale`, `channels`, `lastSeenAt`, `createdAt`, `isCurrent` — explicitly **not** the `token` or `apnsToken` value. Confirmed by reading the route + the test fixtures. Worth a maintainer eyeball on whether `appVersion` / `locale` reveals more about a paired device than the user expected for the settings-list use case (probably fine — these are user-set, not server-derived).

- **Idempotency cache replay does not re-execute auth.** `withIdempotency` at `src/lib/idempotency.ts:215-216` returns the cached response immediately on hit, before the handler (and therefore before `requireAuth()`) runs. The userId-resolver default at line 149-173 DOES re-validate the cookie session OR the bearer token currently presented, so a revoked token can't read a cached response. But if a user's role changes (USER → ADMIN, or admin demoted) between the original request and the replay, the cached response shape might not match what the user is currently entitled to read. Today none of the wrapped routes vary their response by role (they're per-user data routes), so the impact is zero. Worth a note for future authors who might wrap an admin-aware route in `withIdempotency`.

- **The Coach assistant prose is decrypted into `RecommendationFeedback.recommendationText` on every thumbs rating** (`src/app/api/insights/chat/messages/[id]/feedback/route.ts:99-113`). The prose can contain personalized health observations from the user's data. The admin coach-feedback dashboard explicitly never surfaces this column (`src/components/admin/coach-feedback-section.tsx:18` — "admin view never exposes per-user prose"), and there's no read endpoint for the row, so today the prose is only ever read by the user themselves via the existing recommendation-feedback list. If a future feature adds a "review my feedback history" UI, the route author needs to ensure the prose stays gated behind `userId` ownership. Worth a one-line code comment on the persistence call so future readers don't assume `recommendationText` is generic-enough to surface in shared admin views.
